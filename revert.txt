—--------------------------------------------------EXTRA—----------------------------
`git revert <1st_commit_id>` does NOT take repository back to first commit state.
It actually undoes the changes introduced by the first commit itself.
So if first commit created `first.txt`, reverting it will delete `first.txt`.
To restore repository to the state of first commit while keeping history, we need to
revert the commits that came AFTER it.
Example:
* 1st commit → first.txt
* 2nd commit → adds second.txt
* 3rd commit → adds third.txt
Current state = all 3 files.
Correct method:
```bash id="i3h6rk"
git revert <3rd_commit_id>
git revert <2nd_commit_id>
```
After reverting 3rd commit:
* third.txt removed
* repo becomes same as 2nd commit
After reverting 2nd commit:
* second.txt removed
* repo becomes same as 1st commit
Final state:
```bash id="mqgjra"
first.txt
```
History remains preserved because revert creates new undo commits instead of
deleting old commits.
